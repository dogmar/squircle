Slide System Architecture

How should slides be defined and how should transitions work?

A

Declarative Slide Objects (Recommended)

Each slide is a plain object declaring the target state of every visual property. Framer Motion interpolates between states automatically.

const slides = [
  {
    title: "The Rounded Corner",
    body: "CSS border-radius draws a circular arc...",
    graphic: {
      showRounded: true,
      showSuperellipse: false,
      showCorrected: false,
      amount: 1.5,
      showMeasurement: false,
    }
  },
  {
    title: "The Superellipse",
    body: "Now overlay the superellipse...",
    graphic: {
      showRounded: true,
      showSuperellipse: true,
      amount: 1.5,
    }
  },
];

Pros

  • Simple to author & reorder slides
  • Framer Motion handles all interpolation
  • Easy to add/remove slides later

Cons

  • Complex multi-step transitions need careful state design
B

State Machine (XState-style)

Define slides as states in a finite state machine with explicit transitions, guards, and side effects.

const machine = {
  initial: "intro",
  states: {
    intro: {
      graphic: { showRounded: true },
      on: { NEXT: "overlay" }
    },
    overlay: {
      graphic: { showSuperellipse: true },
      on: { NEXT: "mismatch" }
    }
  }
};

Pros

  • Explicit transition logic
  • Could support branching paths

Cons

  • Adds XState dependency or custom machine
  • Overkill for a linear sequence
C

Keyframe Timeline

Define a continuous timeline with keyframes at specific points. Slides are just named positions along the timeline.

const timeline = {
  duration: 5, // total "units"
  keyframes: [
    { at: 0, label: "intro", ... },
    { at: 1, label: "overlay", ... },
    { at: 2.5, label: "sweep", ... },
  ]
};

Pros

  • Supports continuous scrubbing
  • Natural for video-like presentations

Cons

  • More complex to implement
  • Harder to add discrete text changes
  • Overkill for step-based tour
